## Slide 3: 우리 산업의 현실 (2분)

### 상품 기획자의 하루

숙련된 MD 한 명이 패키지 하나를 만드는 데 **2~3일**

그 시간 동안 하는 일:
- 경쟁사 상품 분석 (하나투어, 모두투어, 노랑풍선...)
- 현지 도시별 관광지 조합과 동선 검토
- 항공편/호텔 가용성 확인
- 시즌별 가격 설정과 원가 계산
- SNS 트렌드 반영 (요즘 뜨는 스팟, 드라마 촬영지)

### 그런데 시장은 더 빨리 움직입니다

- 성수기 전에 **수백 개** 상품을 동시에 기획해야 하는 현실
- 트렌드의 수명은 **2~4주** — 기획이 끝나면 이미 식은 트렌드
- 숙련 MD의 이직/퇴직 → **노하우 유실**

> "기획 속도가 곧 경쟁력"인 시대에, 2~3일은 너무 깁니다.

### 그래서 AI 기반 상품 기획 자동화 PoC를 시작했습니다

- 대상: 일본/동남아/유럽 등 **주요 노선 패키지 약 100개**
- 목표: MD가 2~3일 걸리는 기획을 AI가 수분 내에 수행
- 검증: AI 결과물을 현업 MD가 직접 리뷰

## Slide 4: 첫 번째 시도 — Conventional RAG (2분)

### Conventional RAG: 100개 상품 데이터를 AI에게 주고 "만들어봐"

```
100개 패키지 상품 데이터를 문서로 저장
    ↓
"오사카 3박" 요청 → 유사한 상품 문서를 검색
    ↓
검색된 문서를 AI에게 참고 자료로 제공
    ↓
AI가 새 상품 생성
```

### 처음엔 괜찮아 보였습니다

패키지명, 일정 구조, 가격대... 형식은 그럴듯합니다.

**하지만 현업 MD가 결과물을 보면?**

## Slide 5: MD가 지적한 세 가지 문제 (2분)

### 1. "이 관광지는 오사카에 없는데요?"

AI가 **실제로 존재하지 않는 관광지**를 만들어냅니다.
또는 존재하지만 **해당 도시에 없는** 관광지를 배치합니다.

> 검색 결과에 충분한 정보가 없으면 AI는 자기 지식으로 "추측"합니다.
> 이것이 **환각(Hallucination)** — 여행상품에서는 치명적입니다.
> **없는 관광지가 포함된 상품은 판매할 수 없습니다.**

### 2. "교토 일정인데 왜 나라 관광지가 안 나와요?"

MD는 교토 하루 일정에 **인근 나라를 묶는 게 당연한 노하우**입니다.
하지만 AI는 "교토"로 문서 검색만 할 뿐, **교토↔나라가 45분 거리**라는 관계를 모릅니다.

> 도시 간 거리, 관광지와 도시의 연결, 동선의 효율성 —
> 이 **"관계"를 AI가 이해하지 못합니다.**

### 3. "요즘 이 지역에서 뜨는 스팟이 빠졌어요"

SNS 트렌드를 반영하려면 별도 시스템을 처음부터 구축해야 합니다.
기존 상품 데이터와 트렌드 데이터가 **연결되지 않으니까**.

## Slide 6: 근본 원인 — "문서 검색"의 한계 (1분)

### AI에게 주는 데이터의 형태가 문제

| 기존 방식 |
|---|
| "오사카 3박4일 벚꽃 패키지" 전체 텍스트가 **하나의 덩어리** |
| 이 안에 도시, 관광지, 호텔, 항공편이 **뒤섞여** 있음 |

하지만 MD가 상품을 기획할 때 머릿속에 있는 것은:

> **오사카** → 도톤보리, 유니버셜 (관광지)
> **오사카** → 나라, 교토 (인근 도시, 각각 45분/30분)
> **교토** → 킨카쿠지, 후시미이나리 (관광지)
> **오사카** → ○○ 호텔 (온천 있음)
> **시즌: 봄** → 벚꽃 테마

MD의 머릿속은 문서가 아니라 **관계의 네트워크**입니다.

## Slide 7: GraphRAG — MD의 노하우를 구조화하다 (2분)

### 핵심 아이디어: 100개 패키지에서 "관계의 지도"를 만들다

같은 100개 상품 데이터를 **"문서"가 아닌 "관계"**로 재구성했습니다.

```
     관광지 ← 도시 → 호텔
     (500+)     ↑     (도시별 배치)
              패키지
       시즌 ←   ↓   → 항공편
              유사상품    (실제 노선)
                ↓
              트렌드
          (SNS 핫플레이스)
```

100개 패키지에서 추출된 관계:
- 115개 도시, 500개+ 관광지, 도시별 연결 완료
- 도시↔인접 도시 거리 관계
- 상품 간 유사도 (도시 겹침 기반 자동 계산)
- 시즌별 상품 분류
- SNS 트렌드 스팟 → 도시 연결

### 비유: 네비게이션

| Conventional RAG | GraphRAG |
|---|---|
| 여행 가이드북에서 "오사카" 페이지를 찍어서 AI에게 줌 | AI에게 **네비게이션**을 줌 — 도시 간 거리, 관광지 위치, 호텔 위치가 다 연결된 지도 |
| AI는 페이지를 읽고 추측 | AI는 지도를 보고 **길을 찾음** |

## Slide 8: PoC 결과 — 100개 패키지에서 추출한 것 (2분)

### 100개 상품 데이터로 구축한 Knowledge Graph

| 항목 | 규모 | MD 관점에서의 의미 |
|---|---|---|
| 패키지 상품 | 약 100개 | 일본/동남아/유럽/하와이 주요 노선 |
| 도시 | 115개 | 각 패키지가 방문하는 도시 |
| 관광지 | 545개 | 도시별로 분류·연결된 실제 관광지 |
| 호텔 | 도시별 연결 | 어느 도시에 어떤 호텔이 있는지 |
| 항공 노선 | 89개 | 실제 운항 노선만 |
| 상품 간 유사도 | 469쌍 | 도시 겹침 기반 자동 계산 |
| 시즌 연결 | 84건 | 벚꽃/단풍/겨울 시즌별 분류 |
| **총 관계(엣지)** | **약 5,800개** | 100개 상품에서 5,800개의 관계가 도출 |

### 핵심 포인트

> 100개 상품 "문서"를 그대로 주면 AI는 100개 문서만 봅니다.
> 같은 데이터를 "관계"로 재구성하면 AI는 **5,800개의 연결**을 활용합니다.
>
> 특히 도시↔관광지 연결이 **545개** — AI가 각 도시의 실제 관광지를 정확히 알게 됩니다.

## Slide 9: 환각 해결 — Before vs After (2분)

### Before: AI가 만들어냄

```
MD: "교토 하루 일정 짜줘"
AI: "교토 황금 정원, 사쿠라 테라스, 히가시야마 골목길..."
MD: "...이 중 절반은 없는 곳인데요"
```

### After: AI가 Graph에서 찾아옴

```
MD: "교토 하루 일정 짜줘"
AI: (Graph에서 교토 연결 관광지 탐색)
    → "킨카쿠지, 후시미이나리, 아라시야마"
    → 카테고리: 신사 → 신사 → 자연 (다양하게 배치)
    → 가족여행이면 family_friendly 관광지 우선
MD: "바로 쓸 수 있겠네요"
```

### 왜 가능한가?

AI에게 **명시적 규칙**을 부여했습니다:
> "Graph에 등록된 관광지만 사용하세요. 없는 관광지를 만들지 마세요."

PoC에서 구축한 Graph에 **545개 관광지가 도시별로 연결**되어 있으니,
AI는 추측할 필요 없이 **"찾아서 조합"**합니다.

| 일정 작성 | 4시간 | MD |
| 가격 설정 | 3시간 | MD |
| **합계** | **2~3일** | |

### After: AI가 초안을 만들고, MD가 검증/튜닝

| 단계 | 소요 | 담당 |
|---|---|---|
| 조건 입력 (지역/기간/테마/예산) | 5분 | MD |

| 결과 검증 + 미세 조정 | 1~2시간 | MD |

### MD의 역할 전환

- **기존**: 엑셀에서 일정을 처음부터 작성하는 **"제작자"**
- **이후**: AI 결과물을 현업 감각으로 다듬는 **"큐레이터"**

> 숙련 MD의 노하우가 Graph에 구조화되어 있으니,
> 신입 MD도 AI 도구를 활용해 빠르게 역량을 발휘할 수 있습니다.

## Slide 11: 트렌드 반영이 빨라집니다 (1분)

### T&H에서 트렌드 = 매출

- 넷플릭스 드라마 촬영지 → 해당 지역 상품 수요 급증
- 인스타 핫플레이스 → 2~4주 안에 상품에 반영해야 의미 있음
- 시즌 이벤트 (벚꽃, 단풍, 축제) → 미리 준비해야 성수기에 판매

### 기존: 트렌드 반영까지 수주 소요

트렌드 감지 → 상품 기획 → 시스템 반영 → 판매 개시
→ 이미 트렌드가 식은 후

### GraphRAG: 하루면 반영

트렌드 스팟을 Graph에 추가 → 해당 도시에 관계 연결 → AI가 자동으로 일정에 삽입

> "최근 이 지역에서 화제인 ○○를 일정에 포함했습니다"
> → 상품 차별화 포인트가 **자동으로** 생성됩니다.

## Slide 12: 유사도 기반 변형 — 한 상품에서 열 상품으로 (1분)

### T&H의 현실: 같은 노선에 다양한 상품이 필요

오사카 3박4일이라도:
- 가족 테마 vs 허니문 테마
- 쇼핑 중심 vs 문화탐방 중심
- 프리미엄 vs 가성비

### GraphRAG의 유사도 시스템

기존 인기 상품을 **기반(reference)**으로:

> "이 상품과 70% 유사하게" → 도시/호텔은 유지, 관광지만 교체
> "이 상품과 30% 유사하게" → 도시 구성은 유지, 호텔/관광지/테마 전부 교체

Graph에서 **대체 관광지, 대체 호텔을 자동으로 탐색**해서 교체하니
한 번의 인기 상품에서 **다양한 변형 상품을 빠르게 생산**할 수 있습니다.

## Slide 14: T&H 산업 확장 가능성 (2분)

### 여행상품 기획 외에도

**호텔 운영**
- 객실 → 어메니티 → 고객 선호도 → 시즌 수요
- "이 고객이 선호하는 객실 타입과 매칭되는 가용 객실은?"

**항공사 노선 기획**
- 노선 → 공항 → 도시 → 수요 → 시즌
- "이 시즌에 수요 증가가 예상되는 미취항 노선은?"

**MICE (기업행사/컨벤션)**
- 행사장 → 도시 → 호텔 → 케이터링 → 교통
- "200명 규모 행사에 적합한 장소와 주변 숙박 옵션은?"

**OTA 추천 시스템**
- 고객 → 검색이력 → 예약이력 → 관심지역 → 유사고객
- 기존 협업 필터링의 한계를 **관계 그래프 기반 추천**으로 돌파

> **공통점: "엔티티 간 관계"가 비즈니스 로직의 핵심인 영역**

## Slide 15: PoC에서 배운 것 (1분)

### 1. AI 품질은 "모델"이 아니라 "데이터 구조"가 결정합니다

같은 100개 상품 데이터라도,
문서로 주면 AI가 추측하고 — 관계로 주면 AI가 사실에 기반합니다.

### 2. 기존 데이터의 가치를 재발견했습니다

100개 패키지에서 **5,800개의 관계**가 나왔습니다.
"교토 하면 나라도 가야지" — MD의 암묵지가 데이터 안에 이미 들어있었고,
이를 **관계로 명시화**한 것뿐입니다.

### 3. 기존 시스템을 건드리지 않았습니다

예약/정산 시스템은 그대로.
그 위에 **관계 탐색 레이어만 추가**하는 PoC였습니다.

## Slide 17: 핵심 메시지

### "AI에게 가이드북을 주지 말고, 네비게이션을 주세요"

기존 RAG:
> AI에게 상품 문서를 주고 "비슷한 거 만들어봐"

GraphRAG:
> AI에게 **도시-관광지-호텔-노선의 관계 지도**를 주고 "여기서 찾아서 조합해"

**결과:**
- 없는 관광지를 만들어내지 않습니다 — **실제 데이터만 사용**하니까
- 교토 옆에 나라를 넣을 줄 압니다 — **관계를 알고 있으니까**
- 트렌드를 즉시 반영합니다 — **관계 하나만 추가하면 되니까**

> **여행상품의 품질은 AI 모델의 성능이 아니라,
> AI에게 주는 데이터의 구조가 결정합니다.**

### 예상 질문과 답변

**Q: 구축 기간과 비용은?**
A: 100개 패키지 기준 PoC를 2주에 구축했고, 월 운영비는 약 30만원입니다. 전체 상품으로 확장해도 구조는 동일하며, MD 한 명의 생산성이 10배 늘어나는 것 대비 미미한 투자입니다.

**Q: 기존 예약/정산 시스템을 바꿔야 하나요?**
A: 전혀 건드리지 않습니다. 기존 시스템 위에 "관계 탐색 레이어"를 얹는 구조입니다. 기획 도구만 추가되는 것이지, 예약/결제/정산 흐름은 그대로입니다.

**Q: MD가 AI에 대체되는 건가요?**
A: 아닙니다. AI는 초안을 만들고, MD는 현업 감각으로 검증/튜닝합니다. 오히려 MD의 노하우가 Graph에 구조화되어 **조직 자산으로 남게 됩니다.** 숙련 MD가 퇴사해도 노하우가 사라지지 않습니다.

**Q: 데이터는 어디서 오나요?**
A: 기존에 운영 중인 패키지 상품 데이터입니다. PoC에서는 주요 노선 100개 패키지를 사용했는데, 이 100개에서만 5,800개의 관계가 도출되었습니다. 새로운 데이터를 수집하는 것이 아니라, 기존 데이터를 "관계"로 재구성해서 가치를 극대화하는 접근입니다.

**Q: AI 환각이 완전히 사라지나요?**
A: 관광지명, 호텔명, 항공편처럼 팩트로 검증 가능한 정보의 환각은 구조적으로 차단됩니다. 상품 설명문이나 하이라이트 카피 같은 창작 영역에서는 AI의 판단이 개입하지만, 이는 MD가 한 번 읽어보면 바로 수정할 수 있는 수준입니다.
